# Phase 2 — UI Design Contract

> Visual and interaction contract for `GET`/`POST /admin/settings`. Generated by
> gsd-ui-researcher, verified by gsd-ui-checker.

This phase adds ONE page to an existing, already-styled admin app. There is no design
system choice to make — the contract below is **pattern-mirroring** of
`app/templates/editor.html` / `app/templates/login.html` and their shared
`app/static/editor.css`, extended only where this phase introduces something genuinely
new (inline per-field validation errors). Every token, class, and copy string cited here
already exists in the repo unless explicitly marked **NEW (this phase)**.

## Design System

| Property | Value |
|----------|-------|

## Color

**Pre-populated from `app/static/editor.css:root` (60/30/10 already established).**

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `--ink` `#0a0c14` | Page background, input backgrounds |
| Secondary (30%) | `--ink-raised` `#10131e` / `--paper-warm` `#11141f` | `.theme-group` fieldset surfaces (settings groups), `.toast`, `.picker` |
| Accent (10%) | `--red` `#c0192c` | See "Accent reserved for" below |
| Destructive | `--red` / `--red-on-ink` `#e84a5e` (same hue family, not a second color) | `.btn--danger` — this app deliberately reuses brand red for destructive actions, distinguished by **outline vs filled treatment**, not by a separate hue. This phase does not introduce a second semantic color. |

Accent reserved for (explicit list — nothing else may use `--red`/`--red-on-ink`):
- The primary **Save** button (`.btn.btn--accent`)
- Focus rings on inputs/selects/buttons (`:focus-visible { outline: 2px solid var(--red); }`)
- The success-state `.toast[data-kind="ok"]` border
- Anchor/link text (`--red-on-ink`)
- The owner-only `⚙ Ajustes · Settings` link is **ghost-styled** (`.btn--ghost`, no red) so
  it does not compete with the Save CTA — matches the existing `Salir · Sign out` link.

Note: `--red-on-ink` is ALSO used for the inline field-error text and error-state
`.toast[data-kind="error"]` border (see Copywriting Contract) — this is the existing
error convention (`.upload-state[data-kind="error"]`), not a new color decision.

## Copywriting Contract

Bilingual ES–EN, single string per D-13, matching the existing `_FORBIDDEN_COPY` /
`_PUBLISH_SUCCESS_COPY` house style (`em dash` or `·` separator, Spanish first).

| Element | Copy |
|---------|------|
| Primary CTA | **"Guardar ajustes · Save settings"** (idle) / **"Guardando… · Saving…"** (in-flight, mirrors `editorApp`'s `saving` pattern) |
| Owner-only dashboard link (D-12) | **"⚙ Ajustes · Settings"** — ghost button, placed on `editor.html`'s topbar between the danger unpublish button and the `Salir · Sign out` link |
| Owner-forbidden (403, PANEL-01) | **"Solo el propietario puede acceder a esta página. — Only the owner can access this page."** (pre-populated from `02-RESEARCH.md` Pattern 1 — reuses `login.html`'s `.forbidden` block shape, rendered via the existing `_auth_html_or_json` 403 → `login.html` fallback, NOT a new template) |
| Success banner (POST 200) | **"Ajustes guardados. — Settings saved."** — rendered via the existing `.toast[data-kind="ok"]` component |
| Error banner (POST 422, any field invalid) | **"Revisa los campos marcados. — Check the highlighted fields."** — `.toast[data-kind="error"]`; accompanies (not replaces) the per-field messages below |
| Per-field inline error (D-03/D-05, `{errors: {KEY: reason}}`) | Render `SettingRejected.reason` **verbatim** (already bilingual per Phase-1's `_SCHEMA` validators) directly under the offending field, using **NEW** `.field-error` class (see Registry/New Elements below) |
| Empty state | **Not applicable.** `settings.all_for_ui()` always returns all 19 configured tunables, pre-grouped; there is no zero-data state for this form (unlike the editor's `.empty-hint` for zero blocks). |
| Destructive confirmation | **None in this phase.** The settings panel only edits validated values within schema bounds — there is no delete/reset action for tunables (unlike the editor's "Restablecer tema · Reset theme" inline-confirm, which belongs to Phase-pre-existing editor scope, not this phase). If a future phase adds a reset-to-default action, it must reuse the existing `.confirm-inline`/`x-data="{confirm:false}"` two-step pattern already in `editor.html`. |

## Field Rendering Contract (D-06/D-07/D-08 → concrete markup)

Pre-populated from CONTEXT.md decisions + RESEARCH.md's `all_for_ui()` metadata
extension (D-09). Each of the 7 `type_tag`s maps to exactly one rendering, reusing
existing `.field`/`.label`/`.field-hint` classes:

| `type_tag` | Control | Notes |
|------------|---------|-------|
| `snowflake` | `<input type="text" inputmode="numeric" pattern="\d{17,20}">` | Placeholder `"123456789012345678"`; `.field-hint`: `"ID de Discord (17-20 dígitos) · Discord ID (17-20 digits)"` |
| `role_list` | Single `<input type="text">`, comma-separated (D-07) | Placeholder `"123, 456"`; `.field-hint`: `"IDs de rol separados por comas · Comma-separated role IDs"` |
| `int_range` | `<input type="number" min="{min}" max="{max}">` (bounds from D-09's `min`/`max`) | `.field-hint` interpolates bounds: `"Entre {min} y {max} horas · Between {min} and {max} hours"` |
| `timezone` | `<select>` populated from D-09's `options` (`zoneinfo.available_timezones()`, sorted), current value pre-selected | One instance only (`REMINDERS_TZ`) |
| `free_string` | `<input type="text">` | `.field-hint` = the schema's existing `hint` value verbatim (e.g. `WHISPER_MODEL`/`OLLAMA_MODEL` host-availability copy) |
| `url` | `<input type="url">` | Matches `editor.html`'s existing social/link URL inputs |
| `lang` | `<select>` with fixed `es`/`en` options, labeled exactly like `editor.html`'s `#f-lang` (`"Español · Spanish"` / `"English · Inglés"`) | Mirror verbatim, do not reword |

Every field wrapper is a `.field` div with `<label class="label">{label}</label>`
(the D-09 `label` metadata) above the control and `.field-hint` below it (if `hint`
present) — identical shape to `editor.html`'s existing fields.

## Layout Contract

- Page chrome: same `.topbar` (sticky, wordmark left, actions right) as `editor.html`.
- Groups render as a vertical stack of `<fieldset class="theme-group">` (reusing the
  existing `.theme-panel { display:grid; gap: var(--sp-sm); }` wrapper), one per
  feature group (`gallery`, `reviews`, `reminders`, `jinxxy`, `meetings`, `forum` — order
  comes from `all_for_ui()`, do not re-sort).
- Each group: `<legend class="label">{group name, bilingual}</legend>` then its fields
  in document order from the schema.
- No two-pane layout (no live preview needed — this is a plain settings form, not the
  editor's edit/preview split). Single column, max-width matching `.preview-col`'s
  640px reading width for form comfort, centered.
- Save button: sticky in the `.topbar`, same position/style class as `editor.html`'s
  `.btn.btn--accent` publish button (`Depends(require_owner)` gates the route, not the
  button — the button is always visible once the page renders, since only the owner
  ever reaches it).

## New Elements (this phase only)

Inline per-field validation errors do not exist anywhere in the current codebase
(the editor app only ever shows top-level toast errors). This is the one genuinely new
CSS surface this phase adds — append to `app/static/editor.css`, do not fork a new
stylesheet (per `02-RESEARCH.md`'s Open Question 1 recommendation):

```css
/* ── Settings panel: inline per-field errors (NEW, Phase 2) ──────────────────── */
.field--invalid input, .field--invalid select {
  border-color: var(--red-on-ink);
}
.field-error {
  margin: 2px 0 0;
  font-family: var(--font-body);
  font-size: 12.5px;
  line-height: 1.45;
  color: var(--red-on-ink);
}
```

Alpine binds `:class="errors.KEY ? 'field--invalid' : ''"` on the `.field` wrapper and
renders `<p class="field-error" x-show="errors.KEY" x-text="errors.KEY"></p>` directly
after the input (before `.field-hint`, so the error is the first thing read after the
field, hint stays as static help text below it).
