# Phase 4 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

**Visual source of truth:** This phase does NOT introduce a new visual language. It (a) migrates
`app/templates/settings.html` off its v1 `editor.css` chrome onto the Phase-3 dashboard shell
(`app/static/dashboard.css`, `_dashboard_base.html`, `_sidebar.html`, `.mod-hdr` pattern — see
`.planning/phases/03-dashboard-shell-tiered-access/03-UI-SPEC.md`, approved), and (b) adds a small,
closed set of NEW components for name resolution (resolved-name preview, role chips, unavailable
marker, cache-cold banner) built strictly from the tokens Phase 3 already locked. No new size,
weight, spacing value, or general-purpose color may be introduced — every new element below maps
onto an existing declared token.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none — same stack as Phase 3 (FastAPI + Jinja2 + Alpine.js, server-rendered, no build step). shadcn gate does not apply. |
| Preset | not applicable |
| Component library | none — hand-rolled CSS. Extend `app/static/dashboard.css` with a `settings` block (field/group/chip/banner styles) ported in spirit from `app/static/editor.css`'s `.field`/`.theme-group`/`.chip` patterns but re-skinned onto dashboard tokens. `app/templates/settings.html` is rewritten to `{% extends "_dashboard_base.html" %}` (drops its own bespoke `<header>`/topbar, which duplicates `_dashboard_base.html`'s). |
| Icon library | none — plain Unicode glyphs, consistent with the sidebar/module-header convention. New glyphs this phase: channel-type cue `#` (text — reuses the existing `#name` display convention, no extra glyph needed), `🗨` (forum), `🔊` (voice); cache-cold banner glyph `⏳`. No icon for roles — the role cue is a color bar/dot (see Color), not a glyph. |
| Font | Inter 400/600 (unchanged, reused from `_dashboard_base.html`'s `<link>`). `--font-mono` (`'Cascadia Code','JetBrains Mono',monospace`, already declared in `dashboard.css` by Phase 3 in anticipation of this exact use) is used for every rendered raw Discord ID — resolved-field ID line, chip ID line, and the unresolved-ID fallback state. |

**Scope note:** SETT-01 requires "no loss of functionality" — every field type
(`snowflake`/`role_list`/`int_range`/`timezone`/`free_string`/`url`/`lang`), the two-pass
validate-then-write flow, per-field inline errors, and the owner-only gate carry over unchanged
from `app/templates/settings.html`/`app/main.py`. Only the CSS skin, page chrome (topbar/sidebar
via `_dashboard_base.html`), and the `snowflake`/`role_list` field templates (extended with name
resolution) change this phase.

## Typography

No new size/weight combination is introduced. Every Phase-4 element maps onto one of the 4 roles
Phase 3 already closed:

| Role | Size | Weight | Line Height | Phase-4 usage |
|------|------|--------|-------------|----------------|
| Label | 12px | 600 semibold | 1.2 | Field labels (existing), raw Discord ID text (font-family overridden to `--font-mono`, NOT uppercase — the uppercase transform stays reserved for table headers/badges), "name unavailable · no encontrado" marker, cache-cold banner heading |
| Body | 13px | 400 regular | 1.5 | Resolved channel/role name text, role-chip name text, cache-cold banner body copy, existing `field-hint`/toast copy |
| Heading | 18px | 600 semibold | 1.2 | Settings module-header title (`.mod-hdr .t`, "Ajustes · Settings") |
| Display | 26px | — | — | Not used — Settings is a `.mod-hdr` module page, not a `.ph` page-title page (matches the existing `module_stub.html` pattern, not `overview.html`'s) |

**Explicit mapping rule (for the checker):** the raw-ID line under a resolved name/chip is Label
role with `font-family: var(--font-mono)` substituted for Inter — size and weight (12/600) are
unchanged from the Label token. This is a font-family override on an existing role, not a new
type combination.

## Color

Reused verbatim from the Phase-3 contract:

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#0f1115` (`--color-bg`) | Page background |
| Secondary (30%) | `#161a21`/`#1d222b`/`#252b36` | Group cards, chip background, form inputs |
| Accent (10%) | `#4f7cff` (`--color-primary`) | Save button, input focus ring — Phase 4 introduces NO new use of this token |
| Destructive | `#f0435a` (`--color-danger`) | Field-invalid border/text only (existing `.field--invalid` pattern) — Phase 4 has no delete/destructive action |
| Settings section accent | `#8b93a3` (`--accent-settings`) | Settings `.mod-hdr` left border + icon color (this page's module identity, per the Phase-3 per-module accent system) |
| Warning/stale | `#ffb020` (`--color-warning`) | Cache-cold banner left border + icon tint (semantic "stale/pending", already declared by Phase 3) |

**Third accent category — per-item resolved-role color (new this phase, distinct from both the 10%
interactive accent and the per-module section-accent system):** each Discord role carries its own
arbitrary color, pushed by the bot cache (D-08). This is DATA, not a design token — it is never
used as a fill, never competes with the 10%/30%/60% budget, and is rendered ONLY as:
- a 3px left color bar on a role chip (same visual mechanism as the per-module `.stat`/`.mod-hdr`
  accent bar — a thin identifying stripe, not a background fill)
- an optional small (8px) color dot, if the executor needs a second placement (e.g. inline next to
  a single resolved role reference) — same rule: bar/dot only, never a fill or text color.

Channel-type cues (text/forum/voice) are NOT colored — they use `--color-text-muted`, identical to
every other icon glyph in the shell. Only roles get a dynamic color; channels do not (no
`snowflake` field in this schema resolves to a role — see Component Notes).

Accent (10%) reserved for: unchanged from Phase 3 — Save button, input focus ring only. Role colors
and the settings section accent are explicitly NOT part of this budget (documented above to avoid
the checker flagging "multiple accent colors" as a violation — they are semantically distinct
systems, exactly as Phase 3 established for per-module accents).

## Phase 4 Component Notes

Supplementary detail beyond the template fields, for planner/executor use (source: CONTEXT.md
D-01–D-09 + Phase 3's locked dashboard tokens).

### Settings page chrome (SETT-01 migration)
- `settings.html` becomes `{% extends "_dashboard_base.html" %}` — drop its bespoke `<header>`
  entirely; the shared topbar (`_dashboard_base.html` lines 18-23) already provides
  "Volver al editor" and "Salir" links.
- Add a `.mod-hdr` block at the top of `{% block content %}`, matching `module_stub.html`'s shape:
  icon `⚙`, title "Ajustes · Settings", description e.g. "Ajustes del bot, agrupados por función ·
  Bot tunables, grouped by feature", `--acc: var(--accent-settings)`.
- Each schema group (`gallery`, `reviews`, `reminders`, `jinxxy`, `meetings`, `forum`, `access`)
  renders as one `.card` (not the old bare `<fieldset class="theme-group">`) — `.card-hdr h3` for
  the bilingual group legend (existing `groupLabel()` map, unchanged), fields stacked inside at
  `md` (16px) gaps.
- **Access group placement (Claude's discretion, resolved):** keep it LAST, after Forum — this
  matches the existing `core/settings.py` `_SCHEMA` declaration order (access entries are appended
  after forum, lines 297/303) and requires no schema reordering. It is also the most
  access-sensitive group, and placing it last avoids drawing the eye away from routine tunables.
- Save button: single page-level `.btn` (not per-group), same placement convention as
  `overview.html`'s CTA area — top-right of the `.mod-hdr` row or immediately below it; either is
  acceptable, but there is exactly ONE save action for the whole page (D-01, unchanged from
  existing behavior).

### Resolved Snowflake Field (SETT-02, D-04, D-09)
Replaces the plain `<input>`-only `snowflake` template. Structure, top to bottom, inside the
existing `.field` wrapper:
1. Field label (existing, unchanged — Label role).
2. **Resolved-name preview row** (new): `[type-icon] name-text` — type-icon is `#`/`🗨`/`🔊`
   per channel kind (`--color-text-muted`), name-text is Body role (13px/400) in full-bright
   `--color-text`. Bound to Alpine via D-09's live client-side lookup — updates on every keystroke
   in the raw-ID input below, before save.
   - **Resolved:** icon + name as above.
   - **Unresolved, cache fresh (D-06):** icon omitted, name-text replaced by the raw ID in
     `--font-mono` + the "Name unavailable · no encontrado" marker (Label role, muted) inline
     after it.
   - **Unresolved, cache cold/stale (D-07):** icon omitted, show ONLY the raw ID in
     `--font-mono`/muted — no marker, no red/broken styling. The section-level cache-cold banner
     (below) carries the explanation; do not repeat it on every field.
3. **Raw ID line** (D-04 "raw ID beneath"): the numeric ID in `--font-mono`, Label size, muted —
   shown even when resolved, directly under the name, so the owner can always see exactly which ID
   is set without opening the input.
4. The existing raw-ID `<input>` (unchanged edit control, D-07/phase-boundary: no dropdown picker
   this phase).
5. Existing `.field-error` / `.field-hint` (unchanged).

No `snowflake` field in this schema resolves to a role (all 6 are channel IDs — gallery, reviews,
jinxxy-announce, meetings-forum, forum, encoding) — the role-color swatch (D-04's other clause)
only ever applies to `role_list` chips below, never to a single `snowflake` field.

### Role Chips (SETT-02, D-05)
Replaces the plain comma-separated `<input>`-only `role_list` template's READ-side rendering (the
edit control itself stays the same single comma-separated raw-ID `<input>`, per D-05 — chips are
additive, rendered above the input as a live preview, same D-09 live-update rule as snowflake
fields):
- One chip per role ID present in the field's current value (live, Alpine-bound).
- Chip shape: rounded rect (`--radius-md`, not full pill — it holds two stacked text lines),
  background `--color-surface-2`, **3px left color bar tinted with that role's actual Discord
  color** (see Color — third accent category, bar only, never a fill).
- Chip content, two lines: `@RoleName` (Body, 13px/400, full-bright) then the raw ID
  (`--font-mono`, Label size, muted) — same stacked pattern as the resolved-snowflake field, for
  visual consistency across both field types.
- Unresolved role in the list: same fallback as snowflake — "Name unavailable · no encontrado"
  when cache is fresh; raw-ID-only (no marker) when cache is cold/stale.
- Chip row: `flex-wrap`, `gap: 8px` (sm) between chips.
- Empty value: render "Sin roles asignados · No roles assigned" (Body, muted) instead of an empty
  chip row.

### Cache-Cold Section Banner (D-07)
- Renders once, above all groups (below the `.mod-hdr`), when the name-cache freshness signal
  (D-08) indicates the bot hasn't pushed recently — same "cold start" condition that must NOT
  trigger per-field unavailable markers.
- Shape: single-line-height banner card, left border `--color-warning` (3px, matching the
  `.mod-hdr`/`.stat` accent-bar mechanism), `⏳` icon, heading (Label role) + body (Body role) per
  Copywriting Contract.
- Disappears automatically once the cache freshness signal reports fresh (Alpine-reactive, same
  data source powering D-09's live lookup).

### What does NOT change this phase
- `int_range`/`timezone`/`free_string`/`url`/`lang` field templates: unchanged, just re-skinned
  onto dashboard `.field`/input tokens (no new component).
- Two-pass validate-then-write POST flow, `require_owner` gate, toast mechanics: unchanged
  (D-03).
- No dropdown/picker UI (FUT-01, explicitly deferred) — raw-ID text input remains the only edit
  control for every `snowflake`/`role_list` field.
