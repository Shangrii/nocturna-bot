<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Discord gateway → bot process | Channel/role names arrive via the non-privileged GUILDS intent into the bot's in-memory cache; the cog reads that cache, never Discord REST. |
| bot process → shared sqlite | The cog is the SOLE writer of discord_names; the app never writes it. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-01 | Spoofing / Elevation of Privilege | name resolution source | mitigate | The cog reads names ONLY from `self.bot.get_guild(GUILD_ID).channels/.roles` (in-memory gateway cache) — zero REST, zero token, zero new scope/intent; byte-identical in trust to HeartbeatCog. No `import discord` or credential is added to the app process. |
| T-04-04 | Tampering | displayed-name integrity | mitigate | The cog is the sole writer; each push is a full-snapshot replace (Plan 01) so stale/deleted entries cannot linger and no partial state is exposed. |
| T-04-05 | Denial of Service | cold-start race / write failure crashing the loop | mitigate | `_push` returns early when `get_guild` is None; the write is wrapped in try/except + `log.exception` so a transient WAL lock (INFRA-02 not yet shipped) never crashes the loop; cadence kept at minutes (low-frequency writer, Pitfall 5). |
| T-04-SC | Tampering | npm/pip/cargo installs | accept | No package installs this phase — discord.py 2.7.1 already pinned; nothing to verify. |
</threat_model>
