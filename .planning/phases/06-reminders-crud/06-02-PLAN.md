<objective>
Add the optimistic-concurrency race guard and pause state to the shared `reminders` table at
the `core/db.py` layer: migrate `paused` and `version` columns onto any existing production
`bot.db` via the established `ALTER TABLE ... ADD COLUMN` idiom, thread an `expected_version`
kwarg through `update_reminder`/`delete_reminder`/`set_next_fire` (conditional
`WHERE id = ? AND version = ?`, `version = version + 1` bump, return `bool`), add
`AND paused = 0` to `due_reminders`, and grow `_REMINDER_UPDATABLE` to include `paused`.

Purpose: REM-03 requires that a panel edit is never clobbered by the scheduler's write-back,
proven under a concurrent-edit test. The version column makes that race deterministic to test
(no threads). REM-02 requires pause to actually suppress firing — which the `due_reminders`
filter delivers. This plan is the persistence foundation both writers (Plan 03 scheduler,
Plan 04 panel) build on.
Output: modified `core/db.py`, new `tests/test_db_reminders_crud.py` including the LOCKED D-17
proof.
</objective>

|----------|-------------|
| bot scheduler ↔ FastAPI panel (shared sqlite) | Two OS processes write one physical table; the version column is the only integrity guard against a lost update. |
| client → panel → update_reminder(**fields) | Column names in `**fields` are attacker-influencable; the allowlist is the tampering guard. |

| Threat ID | Category | Component | Disposition | Mitigation Plan |

| T-06-04 | Tampering | SQL-injection-by-column-name via update_reminder's `**fields` | mitigate | Existing `_REMINDER_UPDATABLE` allowlist (T-08-03), extended to `paused`; any unlisted key is dropped, never interpolated |
