# Phase 6 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

**Visual source of truth:** This phase does NOT introduce a new visual language. It replaces
`app/templates/module_stub.html`'s placeholder for `/reminders` with a real **table + modal** CRUD
surface, built strictly from the tokens Phase 3 already locked (`app/static/dashboard.css`,
`_dashboard_base.html`, `_sidebar.html`, `.mod-hdr`/`.card`/`table` patterns — see
`.planning/phases/03-dashboard-shell-tiered-access/03-UI-SPEC.md`, approved) and extends the same
sheet in the spirit Phase 4 established (`.planning/phases/04-settings-migration-name-resolution/04-UI-SPEC.md`,
approved). No new size, weight, or spacing value is introduced. This phase DOES introduce several
new component *shapes* that did not exist before Phase 6 (modal, confirm dialog, status badge, row
actions, searchable combobox, chip-input for reactions, live schedule preview) — every one of them
is built from the existing closed token set below, never a new one.

**Typography note (source-of-truth correction):** `dashboard.css`'s live `:root` values were bumped
one step post-Phase-3-ship (owner request — "read too small on a wide monitor"), per the comment at
`app/static/dashboard.css:38-49`. This spec uses the **current shipped values** (13/15/21/32px), not
the original 12/13/18/26px recorded in the Phase 3 UI-SPEC — the token *names*, *count* (4 sizes, 2
weights), and *usage mapping* are unchanged, only the pixel values were scaled.

Accent (10%) reserved for: primary CTA, modal Save button, Edit row-action, input/combobox focus
ring — unchanged scope from Phase 3/4, no new interactive-accent surface introduced.

**Paused status badge is intentionally NOT the warning color.** Pause is a deliberate Manager
action, not a stale/error state — it uses neutral secondary-surface + `--color-text-muted` (the
same "no red/broken styling" principle Phase 4 applied to cold-cache states). Only the imminent-fire
caveat (a genuine "heads up, timing is tight" moment) uses `--color-warning`.

## Phase 6 Component Notes

Supplementary detail beyond the template fields, for planner/executor use (source: CONTEXT.md
D-01–D-18 + Phase 3/4's locked dashboard tokens).

### Reminders page chrome (REM-01)
- **Primary visual anchor:** the `.mod-hdr` icon+title (`⏰` "Recordatorios · Reminders") draws the
  eye first, same shell hierarchy as every other module page — no new visual language. Secondary
  focal point is the "Nuevo recordatorio" CTA (top-right, `--color-primary`), tertiary is the table
  itself.
- Page structure: `.mod-hdr` (accent `var(--accent-reminders)`) → CTA row → `.card` containing the
  table (or the `.empty` state when zero rows).
- **Table sort order (Claude's discretion, resolved here):** ascending by `next_fire_utc` (soonest
  first); paused rows sort to the bottom regardless of their stale `next_fire_utc`, grouped after
  every active row. This keeps the actionable/urgent rows at the top and avoids a stale paused
  timestamp interleaving with live ones.

### Table (D-07, D-08, D-09, D-10)
- Columns, in order: **Name · Schedule summary · Channel · Next fire (relative) · Status ·
  Actions.** Message body is never a column (D-07, rejected).
- Channel column: resolved `#channel` name (Body role) via `discord_names` (D-08); no raw ID shown
  in the happy path. Cache-miss renders the muted placeholder text (`#unknown-channel`) in
  `--color-text-faint`, with the raw ID as a native `title=""` tooltip (D-09) — same graceful-miss
  philosophy as Phase 4's snowflake field, but rendered as a stand-in name rather than a "name
  unavailable" marker, per D-09's explicit wording.
- Status column: badge component (`.status-badge`), pill shape (`--radius-full`), 1px border, Label
  role text — `Active` variant: `--color-success` border/text on `--color-surface-2`; `Paused`
  variant: `--color-border-strong` border, `--color-text-muted` text, same surface.
- Next fire column: relative time string (e.g. "in 3 days," "en 3 días") for active rows; literal
  "—" for paused rows (D-10) — never compute or display a relative time against a stale
  `next_fire_utc`.
- Actions column: three icon-only buttons (`✎` Edit, `⏸`/`▶` Pause/Resume, `✕` Delete), `--space-2`
  gap, ghost-button visual treatment (transparent background, `--color-text-muted` default,
  `--color-text`/`--color-primary` on hover for Edit, `--color-danger` on hover for Delete) — never
  filled/solid buttons in the row (keeps the accent budget on the page-level CTA only).

### Create/Edit Modal (D-11, D-12, D-13, D-14)
- Overlay: full-viewport `rgba(0,0,0,.6)` backdrop over `--color-bg`; modal surface reuses `.card`'s
  visual language (`--color-surface`, `--radius-lg`, `--shadow-lg`) at `--space-6` internal padding,
  max-width ~560px, centered, scrollable body if content overflows viewport height.
- Modal title (Heading role): "Nuevo recordatorio · New reminder" (create) / "Editar recordatorio ·
  Edit reminder" (edit) — plus a small `✕` close affordance top-right (ghost, same visual as row
  Delete icon but without the danger-hover, since closing isn't destructive).
- Fields, in order, full parity with Discord (D-12): Name → Frequency selector (weekly / biweekly /
  monthly / one-off) → the ONE schedule field that frequency requires (D-14, mutually exclusive,
  never show more than one schedule field at a time) → Channel (searchable combobox, D-11) →
  Message body (multiline textarea) → Mention role (optional, searchable combobox, D-11) →
  Reactions (optional emoji chip-input, cap 6, D-12).
- **Searchable combobox (D-11):** text input (same visual as `.settings-card input`) that filters a
  dropdown list (`--color-surface-2` background, `--color-border` border, one `--font-sans` Body-role
  row per option, hover state `--color-surface-3`) populated from `discord_names`; a "use raw ID"
  fallback row/toggle at the bottom of the list switches to a plain typed-ID `<input>` (D-11,
  "typed-ID fallback field for a channel/role not yet in the cache").
- **Live next-fire preview (D-13):** a pill/banner directly below the schedule field, same visual
  shape as Phase 4's `.resolved-name` pill (`--color-surface-2` background, `--radius-full`,
  `--space-2` padding) — `⏱` icon + "Próximo disparo · Next fire" label (Label role, muted) + the
  computed timestamp in team timezone (Body role, full-bright). Recomputes on every relevant
  keystroke/selection change, client-side, reusing the extracted `core/reminder_schedule.py` math
  (planner's integration point) — this spec only defines the visual container, not the computation
  wiring.
- **Reactions chip-input (D-12, cap 6):** same `.role-chip-list`/`.role-chip`-style flex-wrap row as
  Phase 4's role chips, but each chip is a single emoji glyph + a small `×` remove control; an "add"
  input/button sits at the end of the row and disables once 6 chips are present, with the "Reacciones
  ({n}/6)" counter (Label role) shown above the row.
- **Imminent-edit caveat (D-16):** when editing a reminder within the imminent-fire window
  (recommended default: 90 seconds — 1.5× the 60s scheduler tick; exact threshold is
  implementation discretion per CONTEXT.md, this spec fixes only the visual treatment once
  triggered), render a single inline line below the schedule field: `⚠` + Label-role text in
  `--color-warning`, using the locked copy above — never a full banner, never a blocking dialog
  (D-16 explicitly calls for "low-noise").
- Save button: single modal-level `.btn` (Accent/`--color-primary`), bottom-right, disabled while a
  request is in flight (existing `.btn:disabled` pattern, reused).
- Validation: inline per-field errors (`.field-error`, reused verbatim) + validate-then-write
  two-pass flow (D-14, reused from Settings/Phase 2). A partial edit can never persist an
  inconsistent schedule — enforced server-side, surfaced as a field error on the active schedule
  field only.

### Delete Confirm Dialog (D-15, D-17, D-18)
- Smaller modal variant (`.confirm-modal`), same overlay mechanism as the create/edit modal, ~360px
  max-width, no form fields — title (Heading role, reduced container) + body (Body role) + two
  buttons: "Cancelar · Cancel" (ghost) and "Eliminar · Delete" (`--color-danger` filled).
- Imminent variant appends the extra warning line (see Copywriting Contract) directly under the
  body copy, same `⚠`/`--color-warning` treatment as the modal's imminent-edit caveat — visually
  symmetric with D-16, per CONTEXT's explicit "symmetric with delete, low-noise" note.
- This dialog is the ONLY place D-18's accepted risk ("fires once more after delete") surfaces in
  the UI — no additional badge/marker needed elsewhere in the table.

### What does NOT change this phase
- `.mod-hdr`, `.card`, table base styles (`th`/`td`/`tr`), `.empty`, `.toast`, `.field-error`/
  `.field-hint`: unchanged, reused verbatim.
- Sidebar, topbar, `require_manager` gate mechanics: unchanged (existing Phase 3 infra).
- No dropdown/picker UI beyond the combobox specified above — this phase's combobox IS the
  dropdown-picker FUT-01 deferred for Settings' raw snowflake fields; Reminders gets it directly
  because D-11 requires it for a good create/edit experience, not as a FUT-01 reversal.
