<objective>
Build the write-execution side of Phase 7: four new `action_queue` `kind` handlers that
the bot's `ActionQueueCog._dispatch` gains, each re-invoking the EXISTING cog
`_publish`/`_unpublish` verbatim (D-08) so the shipped 🟢-marker idempotency delivers
GAL-02's no-double-publish for free.

Purpose & the load-bearing finding: `GalleryCog._publish`/`_unpublish` and
`ReviewsCog._publish`/`_unpublish` were built for the reaction flow — they NEVER raise
and NEVER return a value; on failure they surface a ⚠️ Discord reaction internally and
`return` silently. The `action_queue` dispatcher decides success/failure by whether the
handler RAISES. A naive wrapper that calls `_publish` and returns `{}` would therefore
report every dispatch as `done`, even a swallowed `GitHubPublishError` — the exact
opposite of D-11. The fix (zero changes to gallery.py/reviews.py): read the 🟢-marker
state via the exported pure `_is_published` BEFORE and AFTER the call, and derive the
queue result from the state TRANSITION.
Output: the four handlers + per-kind unit tests covering fresh success, moot success, and
genuine failure.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| shared sqlite action_queue → bot dispatch | A queued action's `kind`/`payload` is app-supplied; a malicious/stale payload could target an arbitrary message id |
| bot dispatch → Discord/GitHub | Only the bot holds credentials; the handler is the sole execution point |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-05 | Tampering / Elevation of Privilege | dispatch handler message scoping | mitigate | Each handler fetches via `channel.fetch_message(message_id)` against the SPECIFIC `config.PHOTO_CHANNEL_ID`/`REVIEWS_CHANNEL_ID` only — a message id from another channel 404s (IDOR-adjacent guard, matches the reaction handlers). |
| T-07-06 | Repudiation / Integrity (double-publish) | 🟢-marker idempotency | mitigate | Handlers re-invoke the existing cog `_publish`/`_unpublish` whose internal 🟢 check is the sole idempotency gate; a concurrent reaction becomes a no-op (GAL-02). No second flag is introduced. |
| T-07-07 | Tampering (silent failure) | success/failure signal | mitigate | Pre/post `_is_published` state-transition check derives success from the actual 🟢 change, not the (always-None) return value — a swallowed GitHubPublishError raises → action_queue.fail (Pitfall 1). |
| T-07-08 | Tampering (unknown kind) | _dispatch lookup | accept | `_dispatch.get(kind)` returning None already raises ValueError → action_queue.fail (verified, unchanged) — no new code needed; the app-side kind allowlist is added in plan 07-03. |
| T-07-SC | Tampering | package installs | accept | No new package this phase. |
</threat_model>
