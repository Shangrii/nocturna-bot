# Phase 7 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

**Visual source of truth:** This phase does NOT introduce a new visual language. It replaces
`app/templates/module_stub.html`'s placeholder for `/gallery` and `/reviews` with real **Pending |
Published queue** surfaces, built strictly from the tokens Phase 3 already locked
(`app/static/dashboard.css`, `_dashboard_base.html`, `_sidebar.html`, `.mod-hdr`/`.card`/`.btn`
patterns — see `.planning/phases/03-dashboard-shell-tiered-access/03-UI-SPEC.md`, approved) and
extends the sheet the way Phases 4 and 6 did (`04-UI-SPEC.md`, `06-UI-SPEC.md`, both approved). No
new size, weight, or spacing value is introduced. This phase DOES introduce several new component
*shapes* that did not exist before Phase 7 (Pending/Published tabs, gallery thumbnail grid card,
review text card, lightbox, compact per-item inline action status, remove-confirm dialog,
named/anonymous badge) — every one is built from the existing closed token set below.

**Sketch precedent (partially binding):** `.planning/sketches/001-dashboard-shell/index.html`
(`pgGallery`/`pgReviews`, lines 414–452) and `.planning/sketches/themes/default.css` (`.gcard`,
lines 186–193) already sketched a gallery grid card (`.gcard`) and a review card shape, and Phase
3's own UI-SPEC pre-declared `.gcard .b`'s 12px body padding as the ONE locked spacing exception
this codebase carries (`03-UI-SPEC.md` line 55-56, "gallery-card body inner padding"). Phase 7 is
the first phase to actually SHIP that class — this spec reuses the `.gcard` name and the 12px
exception verbatim. The sketch's illustrative 5-star rating on review cards is **NOT** part of this
contract — `cogs/reviews.py`'s real data model carries no star rating (author/text/date/anon-badge
only, CONTEXT.md D-04) — do not render stars. The sketch's "Rechazar/Aprobar" review copy is also
superseded by the locked Copywriting Contract below.

## Spacing Scale

Reused verbatim from the Phase-3/4/6 contract — no new token:

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon-to-label gaps (tab icon, `↗` link), badge internal top/bottom padding, `.btn.sm` vertical padding |
| sm | 8px | Row/card action button gaps, `.btn.sm` horizontal padding, tab gaps, chip-style badge horizontal padding |
| md | 16px | Gallery grid gaps (already declared by Phase 3: "stats/gallery-queue grid gaps" — this phase is that gap's first real consumer), review-card-to-review-card vertical gap, tab-row-to-card-list margin |
| lg | 24px | Card padding/margin-bottom (reused, unchanged), lightbox caption block padding |
| xl | 32px | Main content area padding (inherited from `.main`), lightbox-to-viewport-edge minimum margin |
| 2xl | 48px | Not used this phase |
| 3xl | 64px | Lightbox max-height calculation (`calc(100vh - 2 * var(--space-16))`) |

Exceptions: **12px** (`--space-3`) — reused for `.gcard .b` (gallery card body inner padding), the
exact pre-locked exception from `03-UI-SPEC.md`. No NEW exception is introduced. The sketch's
`.gcard .acts` 6px button-row gap is normalized to 8px (`--space-2`) — 6px is not on the closed
scale and was never declared as a second exception; this phase closes that gap the same way Phase 3
normalized the sketch's unused 700-weight/15px/21px values.

phase) — unchanged scope philosophy from Phase 3/4/6, no decorative or general-purpose use
introduced. Per-module accent (gallery/reviews) reserved for: `.mod-hdr` border + icon (existing),

second accent color as a violation).

**Moot-action success is intentionally the SAME visual state as a real success (D-11).** "Ya estaba
publicada." and "Ya estaba quitada." render with the identical `✓`/`--color-success` treatment as a
genuine publish/remove — never a distinct third color, never a warning tone. Only a genuine dispatch
failure uses `--color-danger`.

## Phase 7 Component Notes

Supplementary detail beyond the template fields, for planner/executor use (source: CONTEXT.md
D-01–D-13 + Phase 3/4/6's locked dashboard tokens + the sketch's `.gcard` precedent).

### Page chrome (GAL-01/REV-01)
- **Primary visual anchor:** the `.mod-hdr` icon+title draws the eye first (same shell hierarchy as
  every other module page — no new visual language). Secondary focal point is the Pending tab's
  card list (the default view, D-06); tertiary is the Published tab, one click away.
- Page structure: `.mod-hdr` (accent `var(--accent-gallery)` / `var(--accent-reviews)`) → Tabs row
  → `.card`-wrapped list (gallery grid or review-card stack), or the `.empty` state when the active
  tab has zero rows.
- Both `/gallery` and `/reviews` share this exact structure; only the card body content (image grid
  vs. text cards) and per-module accent/copy differ.

### Tabs (D-06)
- Two tabs per module, horizontal, `Pendientes · Pending` default-active, `Publicadas · Published`
  secondary — a simple `<button>` row (`.tabs` flex container, `--space-2` gap between tabs,
  `--space-4` margin-bottom before the card list).
- Active tab: full-bright `--color-text`, 2px bottom border in the module's own per-module accent
  (`--accent-gallery`/`--accent-reviews`) — same accent-as-identity rule the `.mod-hdr` border
  already uses on this page, not the general 10% interactive accent.
- Inactive tab: `--color-text-muted`, transparent bottom border, hover → `--color-text`.
- Pending tab carries a small numeric count pill (`.tab .cnt`, reusing `.status-badge`'s pill shape
  neutrally — `--color-surface-2` background, `--color-text-muted` text, no semantic color) showing
  the live pending count from the push cache. Published tab carries no count (not the actionable
  queue).
- Switching tabs is a pure client-side Alpine toggle (`activeTab: 'pending' | 'published'`) — both
  row sets are already present in the page payload (D-01/D-02 push-cache read), no extra fetch.

### Gallery grid + card (GAL-01/02/03, D-03, D-05, D-07)
- Grid container `.gq` (reusing the sketch's declared class name): `display:grid;
  grid-template-columns: repeat(auto-fill, minmax(220px, 1fr)); gap: var(--space-4);` — responsive,
  no fixed breakpoint needed (auto-fill collapses column count naturally down to 1 on narrow
  viewports).
- Card `.gcard` (reusing the Phase-3-locked class name/exception): `--color-surface-2` background,
  1px `--color-border`, `--radius-md`, overflow hidden.
  - `.gcard .thumb`: the Discord CDN thumbnail image, `height: 180px; width: 100%; object-fit:
    cover;`, `loading="lazy"` — refreshed every push-cache cycle so the signed URL never goes stale
    in-queue (D-02). Clicking the thumb opens the lightbox (D-07) — `cursor: pointer`, no visible
    affordance needed beyond the cursor (image-click-to-expand is a well-understood pattern).
  - `.gcard .b` (body, **the locked 12px exception**, `padding: var(--space-3)`):
    1. Caption text (Body role) — omitted entirely if empty (see Copywriting Contract).
    2. Meta line (Label role, muted): `{poster} · {relative time}` — poster resolved via the Phase-4
       `discord_names`/push-cache poster field (D-03); no cache-miss fallback UI needed this phase
       beyond showing the raw name the cache provides (poster names come from the bot's own
       display-name push, not a separately-resolved snowflake).
    3. `↗ Abrir en Discord · Open in Discord` link (Label role, muted, hover `--color-text`) — opens
       the Discord message in a new tab (`target="_blank" rel="noopener"`).
    4. Actions row (`.gcard .acts`, `--space-2` gap, normalized from the sketch's 6px): on the
       **Pending** tab, a single "✓ Aprobar · Approve" `.btn.sm` (Accent). On the **Published** tab,
       a single "✕ Quitar · Remove" `.btn.sm.danger`. Never both buttons on the same card — the
       tab you're viewing determines which single action is available (mirrors the reaction flow:
       ✅ only makes sense on a not-yet-published message, 🌙 only on a published one).
    5. Inline action-status chip (see below), rendered in place of the actions row once a click has
       fired, until the item's status resolves.
- **New button size modifier `.btn.sm`:** `padding: var(--space-1) var(--space-2); font-size:
  var(--text-xs);` — reuses only already-declared tokens (4px/8px/13px), needed because `.btn`'s
  default padding is too large for a card-width action row. `.btn.sm.danger` composes with the
  existing `.btn.danger` background override.

### Lightbox (D-07)
- View-only — no Approve/Remove buttons duplicated here (single action source of truth stays the
  card, avoiding a second click-path that could double-submit against the action queue).
- Reuses the existing `.modal-overlay` mechanism (full-viewport `rgba(0,0,0,.6)` backdrop,
  `@keydown.escape.window` + `@click.self` to close, same as the Reminders modal/confirm dialogs).
- Content: the full-size image, `max-width: min(92vw, 1100px); max-height: calc(100vh - 2 *
  var(--space-16));`, centered, plus a small caption line (Body role, muted) below if present, and a
  `✕` close button top-right (same ghost visual as the Reminders modal's close, `.modal-close`,
  reused verbatim).

### Review card (REV-01/02, D-04, D-05, D-13)
- One `.review-card` per review (extends `.card`, not a new surface primitive), stacked vertically,
  `--space-4` gap between cards (the "text cards" of D-05).
- Left accent bar (3px, `--accent-reviews`) — same `.acc-bar` mechanism the sketch used, matching
  the existing per-module identity system.
- Header row: author name (Body role, full-bright) — either the resolved Discord display name for a
  named review, or the literal `Anónimo` (`REVIEW_ANON_LABEL`, D-13, never the real name) — plus the
  named/anonymous badge (`.status-badge`-shaped pill, neutral colors, no success/danger semantics —
  it's metadata, not a state) — plus the date (Label role, muted, right-aligned via `.spacer`).
- Body: the full review text (Body role, muted, quoted — `“{text}”`).
- Actions row + inline status chip: identical mechanics to the gallery card — "✓ Aprobar · Approve"
  on Pending, "✕ Quitar · Remove" on Published, same `.btn.sm` sizing.
- **Anonymity guarantee (D-13, hard constraint for the checker):** no UI element on this card may
  ever render the anonymous submitter's real name/id — the push cache itself never carries it
  (backend guarantee), and this card's author-name binding must read the SAME field the backend
  already discards identity from. This is a data-contract note, not a new visual rule, but is
  called out here because it is load-bearing for REV-01/D-13 compliance.

### Inline per-item action status (D-09, D-11, reused from Phase 5)
- A compact chip that replaces the actions row on the acted-upon card the moment Approve/Remove is
  clicked, using the EXACT same `stateKind()`/`statusCopy()` state machine and copy as
  `overview.html`'s `actionProofApp` (`working` → `offline` → `ok` → `failed`), just sized for a
  card footer instead of the full-width Overview proof card: `display:flex; align-items:center;
  gap: var(--space-2); font-size: var(--text-xs);` — mark color follows the Color table above
  (`--color-success` ok, `--color-danger` failed, `--color-warning` offline, `--color-text-muted`
  working).
- On `failed`, a small "Reintentar · Retry" `.btn.sm.ghost` appears inline next to the status text
  (same retry affordance as Phase 5's Overview proof card, reused verbatim — D-02).
- On `ok` (including the D-11 moot-success case), the chip shows the "✓ Listo · Done" mark plus the
  moot-specific detail line ("Ya estaba publicada."/"Ya estaba quitada.") ONLY when the dispatch
  result flagged it as a no-op; a genuine fresh publish/remove shows just "✓ Listo · Done" with no
  extra line. **Both render with the identical success color/mark** — the distinction is copy-only,
  never a different visual state (D-11's core requirement: never a red error for a moot action).
- Short-poll cadence: reuse the Phase-5 `/api/actions/{id}` polling interval unchanged (Claude's
  discretion in CONTEXT resolved to "reuse the Phase-5 value" — no new interval is invented here).

### Remove confirm dialog (D-10)
- Reuses the exact `.confirm-modal` shape Phase 6 built for reminder deletion (~360px max-width, no
  form fields, title + body + two actions) — same overlay mechanism, same `.modal-actions` button
  row: "Cancelar · Cancel" (ghost) + "Quitar · Remove" (`.btn.danger`, NOT the reminder pattern's
  "Eliminar · Delete" label — Remove here is reversible per D-10, so the confirm button and dialog
  copy use "Quitar/Remove," never "Eliminar/Delete," to avoid implying permanent deletion).
- Approve has NO confirm dialog anywhere (D-10) — clicking "✓ Aprobar · Approve" fires the enqueue
  immediately and shows the inline status chip.

### What does NOT change this phase
- `.mod-hdr`, `.card`, `.empty`, `.toast`, `.btn`/`.btn.ghost`/`.btn.danger`, `.modal-overlay`,
  `.confirm-modal`, `.modal-close`: unchanged, reused verbatim.
- Sidebar, topbar, `require_manager` gate mechanics: unchanged (existing Phase 3 infra).
- **Sidebar pending-count badges are explicitly NOT added this phase.** `03-UI-SPEC.md` anticipated
  `.nav-item .cnt` arriving with Phase 7, but no CONTEXT.md decision (D-01–D-13) authorizes it, and
  D-12 explicitly scopes Phase 7 to "approve/remove parity ONLY." The Pending count instead surfaces
  via the in-page tab badge (`.tab .cnt`, above). Revisit the sidebar badge only via an explicit
  future decision, not as an implicit side effect of this phase.
- Editor-credit and NSFW controls (D-12): no UI ships for either — they remain Discord-only
  (`/galeria creditar`).
- No star-rating UI for reviews (sketch was illustrative only; the real data model has none).
