<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser → POST /api/actions | untrusted `kind`/`payload` from a manager's browser crosses into enqueue |
| browser → GET /api/actions/{id} | a manager reads any action row by numeric id |
| FastAPI app → sqlite only | the app NEVER holds Discord credentials; it only writes/reads action_queue rows |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-07 | Spoofing / broken access control | all 3 /api/actions routes | mitigate | `Depends(require_manager)` on every route — owner OR Manager only, reusing the Phase-3 choke point; fails closed (403) without a session (ASVS V4) |
| T-05-08 | Tampering (arbitrary kind injection) | POST /api/actions | mitigate | `_ALLOWED_KINDS` allowlist checked at the route boundary before enqueue → clean 422; parameterized SQL means `kind` can never alter the query shape (ASVS V5) |
| T-05-09 | Information Disclosure | GET /api/actions/{id} | accept | `action_queue` is one SHARED table all Managers act on (D-09, single flat trust tier); no per-Manager partitioning exists anywhere in the app — deliberate design, not a gap. Payloads carry action params (ids), never tokens/secrets (V6 n/a) |
| T-05-10 | Denial of Service | rapid repeated enqueue | accept | No rate limiting exists anywhere in this internal single-guild staff tool today; consistent with existing posture (slowapi rejected milestone-wide). Not a new gap introduced by this phase |
| T-05-11 | Elevation of Privilege | reintroducing a Discord credential into the app | mitigate | Explicitly NOT done — the app only reads/writes action_queue rows; all Discord side effects stay bot-process-only (PITFALLS Pitfall 4) |
| T-05-SC | Tampering (supply chain) | pip installs | accept | No new packages this phase; no install task |

No `high`-severity threats: access-controlled, allowlisted, no secrets on the wire.
</threat_model>
