core/action_queue.py public surface (all wrapped in @_retry_on_locked except get_status):
  enqueue(kind: str, payload: dict, requested_by: str) -> int          # inserts status='pending'
  recover_stale_claims() -> int                                        # claimed>threshold -> pending
  claim_next() -> sqlite3.Row | None                                   # oldest-first, status->'claimed'
  complete(action_id: int, result: dict) -> None                      # status='done' + _purge_terminal
  fail(action_id: int, error: str) -> None                            # backoff requeue OR terminal 'failed'
  retry(action_id: int, requested_by: str) -> int | None              # fresh row from a 'failed' row only
  get_status(action_id: int) -> sqlite3.Row | None                    # read-only, NOT retry-wrapped
  _retry_on_locked(fn)                                                 # module-internal decorator; unit-tested from tests/test_action_queue.py

Module constants (RESEARCH.md Pattern 1): _LOCK_RETRY_DELAYS=(0.05,0.15,0.4);
  _MAX_DISPATCH_ATTEMPTS=3; _BACKOFF_SECONDS=(2,5,15); _STALE_CLAIM_SECONDS=60.

action_queue table (RESEARCH.md Pattern 1 DDL): id INTEGER PK AUTOINCREMENT, kind TEXT,
  payload_json TEXT, status TEXT DEFAULT 'pending' (pending|claimed|done|failed), result_json,
  error, requested_by TEXT, requested_at TEXT, claimed_at, completed_at,
  attempts INTEGER DEFAULT 0, next_attempt_at TEXT; + INDEX idx_action_queue_status(status).

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| caller → action_queue write functions | `kind`/`payload`/`requested_by` reach parameterized SQL |
| both processes → shared sqlite file | bot + app write the same `action_queue` table concurrently |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-01 | Tampering | `enqueue`/`claim_next`/… SQL | mitigate | Parameterized SQL only (`?` placeholders); `kind`/`payload` never interpolated into query text, so no column/SQL injection regardless of caller input |
| T-05-02 | Denial of Service | writer/writer sqlite contention | mitigate | `PRAGMA busy_timeout=8000` (this plan) + `_retry_on_locked` on all queue write paths; proven by the D-12 gate in Plan 04 |
| T-05-03 | Repudiation / data loss | `_purge_terminal` keep-last-N | mitigate | Purge scoped to `status IN ('done','failed')` so a pending/claimed row can never be silently dropped (D-08 invariant; Pitfall 1 regression test) |
| T-05-SC | Tampering (supply chain) | pip installs | accept | No new packages this phase (RESEARCH Package Legitimacy Audit: N/A) — stdlib + already-installed deps only; no install task exists |

No `high`-severity threats: all data-tier, parameterized, single-guild internal.
