<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| caller → settings.set | Untrusted-shaped value crosses into persistence. In Phase 1 the only caller is seed_defaults() (trusted .env), but set()'s validators are the control Phase 2's HTTP POST will rely on — they must be load-bearing now. |
| settings table → settings.get | A row could be corrupt/out-of-range (manual DB edit, future schema drift) and is read into a cog. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-02-01 | Tampering | settings.set key → SQL | mitigate | Validate key against _SCHEMA allowlist BEFORE any SQL; pass key and value only as parameterized `?` placeholders — never f-string interpolation (mirrors core/db.py _REMINDER_UPDATABLE). |
| T-01-02-02 | Tampering | settings.set value (V5 Input Validation) | mitigate | Per-type validators reject bad snowflake / interval / TZ / lang before persistence; invalid input raises SettingRejected and writes nothing, so get() can never return a cog-breaking value. |
| T-01-02-03 | Denial of Service | settings.get on corrupt/missing row | mitigate | Broad-except fallback to the .env/default seed (STORE-04); a corrupt row degrades to default rather than crashing the bot's read path or its startup fail-fast block. |
| T-01-02-SC | Tampering | package installs | accept (N/A) | No packages installed — core/settings.py is stdlib + core/db.py only (01-RESEARCH.md Package Legitimacy Audit). No install task; nothing to verify. |

Note: this plan introduces NO network-reachable surface — set()'s only writer via HTTP POST is Phase 2 (PANEL-03),
so V2/V3/V4 (auth/session/access-control) are out of scope here.
</threat_model>
